# Insight Agent — 系统架构概览

> 电商数据异常检测与根因分析 Agent。用户上传经营报表 + 自然语言提问 → 自动扫描异常、因果推理、按需深挖 → 输出可执行的分析报告。

---

## 产品定位

Insight Agent 的目标是把「上传数据 → 发现问题 → 定位根因 → 给出建议」这条链路变得更可靠，让数据根因分析门槛进一步降低。

**核心用户场景**：
- 「8 月 GMV 有没有异常？」→ 快速扫描，得出结论
- 「美妆品类转化率怎么了？」→ 多维度下钻，定位异常来源
- 「为什么 8 月整体业绩下滑？」→ 因果推理 + 建议补充数据做深度分析

---

## 核心设计理念：渐进式分析深度

这是整个系统最重要的设计决策。

传统做法是对所有问题跑同一套完整分析流程（扫描 → 下钻 → 归因 → 报告），导致简单问题也要等很久，复杂问题又分析不够深。我们的方案是**让系统自动判断问题需要多深的分析**，然后只做必要的工作：

```
用户提问
  ↓
Commander（理解意图，判断分析深度）
  ↓
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  descriptive（概览型）     "8月GMV有没有异常？"              │
│  → 全维度快速扫描 → 直接出结论                              │
│  ⏱ 2-3 次 LLM 调用，~30s                                   │
│                                                             │
│  diagnostic（诊断型）      "美妆品类转化率怎么了？"          │
│  → 扫描 + 交叉下钻 + 异常定位                               │
│  ⏱ 3-5 次 LLM 调用，~60s                                   │
│                                                             │
│  causal（归因型）           "为什么8月业绩下滑？"            │
│  → 扫描 + 因果推理 + 可选深度分析                            │
│  ⏱ 3-12 次 LLM 调用，~60-180s                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**为什么这样设计？**

| 设计决策 | 备选方案 | 选择理由 |
|---------|---------|---------|
| 按语义自动判断深度 | 用户手动选择分析模式 | 用户不知道自己需要什么深度，且增加操作成本 |
| 统一入口，内部分流 | 前端提供"简单/深度"两个按钮 | v5-v8 曾用 simple/complex 双路径，出现路由错误；统一管线后错误率归零 |
| 宁可多分析一步 | 严格按关键词匹配 | 默认 diagnostic（而非 descriptive），避免用户问"怎么了"却只拿到是/否回答 |

## 系统架构

### 整体流程

```
用户提问 + 上传数据
       ↓
  ┌─ Gateway ─────────────────────────────────────────────┐
  │  校验问题合法性（字段是否存在、时间是否在范围内等）      │
  │  判断输出类型（TABLE / CHART）                          │
  └───────────────────────────────────────────────────────┘
       ↓ VALID                           ↓ INVALID → 提示用户修改
       │
  ┌─ Step 1: Quick Scan ─────────────────────────────────┐
  │  Commander (1次LLM) → 生成扫描计划                     │
  │  Scan Loop (PythonAgent) → L1全局扫描 → L2按维度下钻   │
  │  Detect (纯规则) → 标记异常/正常                       │
  │  Arbiter (0-1次LLM) → 仅在不确定时裁决                │
  └───────────────────────────────────────────────────────┘
       ↓ 按 analysis_depth 分流
       │
       ├── descriptive → 格式化扫描结果 → 报告 → END
       │
       ├── diagnostic / causal
       │     ↓
       │   ┌─ Step 2: Reasoner ──────────────────────────┐
       │   │  因果推理：谁导致了谁？                        │
       │   │  输出：根因 + 因果链 + 信心等级                │
       │   │  判断：是否需要更深入的分析？                  │
       │   └─────────────────────────────────────────────┘
       │         ↓
       │         ├── 信心充足 → 生成报告 → END
       │         │
       │         └── 信心不足 → 展示初步结论，询问用户
       │                          ↓ 用户上传补充数据（可选）
       │                       ┌─ Step 3: Deep RCA ──────┐
       │                       │  合并补充数据             │
       │                       │  全量扫描新增指标         │
       │                       │  假设验证循环             │
       │                       │  综合报告                 │
       │                       └──────────────────────────┘
       │
       └── scan 失败 → 降级到 ReAct 循环（安全网）
```

### 各步骤的 LLM 调用控制

一个关键的产品约束是**成本和速度**。每次 LLM 调用 ≈ 2-5 秒 + API 费用，所以我们严格控制调用次数：

| 环节 | 是否调 LLM | 设计理由 |
|------|-----------|---------|
| Gateway 校验 | ❌ 纯规则 | 字段/时间范围校验不需要 LLM |
| Commander 规划 | ✅ 1 次 | 需要理解用户意图，规则无法覆盖所有表述 |
| L1/L2 扫描执行 | ✅ 每层 1 次 | PythonAgent 生成 pandas 代码 |
| 异常检测 (Detect) | ❌ 纯规则 | \|change_pct\| > 阈值，不需要 LLM 判断 |
| Arbiter 裁决 | ✅ 0-1 次 | 仅在"多异常幅度接近"等不确定场景 |
| Reasoner 因果推理 | ✅ 0-1 次 | 单异常不调（规则即可），多异常才需要 LLM |
| 补充数据扫描 | ❌ 纯 pandas | v9.8 改为数据驱动，避免盲目调 LLM 探索 |
| 报告生成 | ✅ 1 次 | 需要自然语言表达 |

**结果**：descriptive 路径全流程仅 2-3 次 LLM 调用，用户体感 < 30 秒。

## 核心组件详解

### Commander — 分析计划生成器

Commander 是整个系统的"大脑"，负责把用户的自然语言问题转化为结构化的扫描计划（`AnalysisFrame`）。

**它做的关键判断**：

1. **分析深度**（最重要的判断）
   - "8 月 GMV 有没有异常" → descriptive（用是/否就能回答）
   - "美妆品类怎么了" → diagnostic（"怎么了"隐含归因需求）
   - "为什么 GMV 下降" → causal（明确要求追因）

2. **时间窗口推断**
   - "上月" → 月环比（自动推算具体日期）
   - "7 月中到 8 月中" → 自定义窗口（支持模糊表述："月中"→15 日）
   - 所有日期基于数据实际范围推算，不用 `datetime.now()`

3. **指标聚合方式**（从 Schema 自动推断）
   - 支付金额、订单数 → sum
   - 转化率、ROI → weighted_avg（sum(分子)/sum(分母)，而非 mean）
   - 这个判断直接决定计算结果的正确性

4. **问题类别**（v9.7 新增，用于零异常时的智能响应）
   - roi_ad / competitor / inventory / promotion / general
   - 例：用户问"投放效率为什么变差"但扫描没有异常 → 系统知道应该建议补充「投放明细数据」而非笼统说"一切正常"

**兜底机制**：Commander LLM 失败时，降级到纯规则生成扫描计划（从 Schema 推断所有参数），确保流程不中断。

### Quick Scan — 全维度快速扫描

Quick Scan 是 Step 1 的核心，替代了早期版本的逐假设验证模式。

**设计变化（v7 → v8 的关键迭代）**：

| 维度 | v7（旧） | v8+（现在） |
|------|---------|-----------|
| 扫描方式 | 逐维度单独查询 | 一次性全维度扫描 |
| LLM 调用 | 每个维度 1 次 | L1 全局 1 次 + L2 条件触发 |
| 异常检测 | LLM 判断 | 纯规则（\|change_pct\| > 阈值） |
| 下钻逻辑 | 固定下钻 | L1 有异常才触发 L2 |

**扫描层级**：
- **L1（全局）**：对所有数值指标计算当期 vs 上期的变化率，一次性输出
- **L2（下钻）**：仅当 L1 发现异常时触发，按分类维度（品类/渠道/来源）拆解异常来源

**Arbiter（裁判）**：大多数情况下不需要调用，仅处理三种不确定场景：
- 多个异常维度变化幅度接近（差距 < 3%），无法分主次
- L1 有异常但 L2 下钻后全正常（可能是整体趋势而非某分组导致）
- 无异常但多个维度接近阈值

### Reasoner — 因果推理引擎

Reasoner 接收 Quick Scan 的扫描结果，做因果推理并判断分析是否充分。

**按异常数量的差异化处理**：

| 异常数 | 处理方式 | 设计理由 |
|-------|---------|---------|
| 0 个 | 按问题类别差异化响应 | descriptive 直接说"正常"；causal 检测是否缺数据 |
| 1 个 | 规则判定，不调 LLM | 单一异常因果关系明确，LLM 是浪费 |
| 2+ 个 | LLM 因果推理 | 需要判断谁是因、谁是果 |

**零异常的智能响应（v9.7 核心改进）**：

早期版本对"零异常"一刀切返回"一切正常"，但这对 causal 类问题不合适——用户问"投放效率为什么变差"，如果现有数据没异常，应该建议补充投放明细数据，而非说"没问题"。

改进后的逻辑：
- descriptive + 零异常 → 直接返回"各指标正常"（这就是用户想要的答案）
- causal + 零异常 → 检测当前 Schema 是否缺少关键数据（通过 `question_category` 映射），如果缺 → 建议补充并触发 Deep RCA 流程

**`needs_deep_rca` 决策**（是否需要进入 Step 3）：

采用保守策略，默认 `false`，仅当满足以下任一条件时才设为 `true`：
- 信心等级为 low（因果链不确定，现有数据无法解释）
- 多根因变化幅度接近（差距 < 5%），无法区分主次
- causal 类问题且当前数据缺少对应领域的明细数据

### Deep RCA — 两轮交互的深度分析

这是系统中**唯一需要用户参与的环节**，也是一个重要的产品设计决策。

**为什么不全自动？**

Deep RCA 需要补充数据（投放明细、竞品数据、库存表等），这些数据通常不在初始上传的汇总报表里。与其让 Agent 在不完整数据上"猜"，不如：
1. 先告诉用户目前发现了什么、还不确定什么
2. 建议需要补充的具体数据类型（含建议列名）
3. 用户上传后再做深度分析

**两轮请求流程**：

```
第一轮：用户提问 → Quick Scan → Reasoner → "初步结论 + 建议补充数据"
                                              ↓ 用户上传补充数据（可选）
第二轮：合并数据 → 全量扫描新增指标 → 假设验证 → 综合报告
```

**v9.8 的关键改进——数据驱动而非盲探**：

早期版本在 Deep RCA 阶段直接让 LLM 生成假设然后逐个验证，效率低且容易遗漏。v9.8 改为：
1. 先用纯 pandas 对补充数据的新增列做全量快速扫描（0 次 LLM），得到客观的数据发现
2. 基于扫描发现生成假设（"投放花费下降 70%，验证是否导致流量下降"）
3. 扫描发现直接写入证据板，供后续报告引用

## PythonAgent 与 Prompt 设计

### Prompt 分层架构

PythonAgent 负责将自然语言查询转化为 pandas 代码。它的 Prompt 采用四层结构：

```
┌─ Layer 4: Hard Constraints（尾部 = 最高权重）──────────┐
│  不可违反的规则：ratio 必须用 sum(分子)/sum(分母) 等    │
├─ Layer 3: Output Rules ────────────────────────────────┤
│  输出格式：result_df / answer / summary 变量规范        │
├─ Layer 2: Skills（动态注入）───────────────────────────┤
│  由 SkillLoader 根据查询内容匹配加载的领域知识          │
├─ Layer 1: Base ────────────────────────────────────────┤
│  角色定义 + 分析五要素提取方法论                        │
└────────────────────────────────────────────────────────┘
```

**为什么分层？** 两个原因：
1. **维护性**：领域知识修复改 Skill 文件即可，不动主 Prompt
2. **权重控制**：DeepSeek 模型对 System Prompt 尾部内容权重更高，把"绝对不能违反"的规则（如 ratio 聚合方式）放在最后

### Skill 动态注入

SkillLoader 根据用户查询关键词匹配相关的领域知识文件：

| 触发关键词 | 加载的 Skill | 解决的问题 |
|-----------|-------------|-----------|
| ROI、转化率、客单价 | ecommerce_metrics.md | ratio 指标的正确聚合方式 |
| 趋势、同比、环比 | time_analysis.md | 时间窗口计算模式 |
| 平均、标准差、分布 | statistical_methods.md | 统计方法的正确使用 |
| 对比、增长、归因 | comparison_analysis.md | 对比分析的输出格式 |

**准确率提升**：通过 Skill 系统，PythonAgent 的代码生成准确率从 `[51%]` 提升至 `[97.1%]`。其中贡献最大的是 ecommerce_metrics Skill（ratio 类指标的聚合方式是最常见的错误来源）。

> *注：具体准确率数据见 [evaluation/python_agent/accuracy_improvement.md](../evaluation/python_agent/accuracy_improvement.md)*

## 关键设计原则

### 1. Schema 零硬编码

系统不预设任何列名。Commander、Reasoner、Deep RCA 的所有指标名、维度名、聚合方式、异常阈值均从上传数据的 Schema 动态推断。这意味着：
- 换一套完全不同列名的数据，系统无需修改即可工作
- 通过 `unit_hint`（从数据特征自动识别）决定聚合方式，而非靠列名关键词猜测

### 2. 渐进式深度，按需分析

每一步只在必要时才进入下一层：
- L2 下钻仅在 L1 发现异常时触发
- Reasoner 仅在 diagnostic/causal 问题时调用
- Deep RCA 仅在信心不足时建议
- 全流程最少 2 次、最多 12 次 LLM 调用

### 3. Fallback 安全网

任何环节失败都有降级方案，不会让用户看到空白结果：
- Commander LLM 失败 → 纯规则生成扫描计划
- Quick Scan 失败 → 降级到 ReAct 假设验证循环
- 报告缺少"分析局限"段落 → 自动补写

### 4. 两轮 Human-in-the-loop

Deep RCA 是唯一需要用户介入的环节。设计考量：
- AI 不应该在数据不足时"编造"根因
- 用户对自己的数据最了解，知道哪些补充数据可获取
- 明确告诉用户"我还不确定什么"，比给出低信心的结论更有价值

## 技术栈

| 组件 | 技术选型 | 选择理由 |
|------|---------|---------|
| 状态机编排 | LangGraph | 支持条件路由、子图、状态传递 |
| LLM | DeepSeek Chat + Reasoner | Chat 用于代码生成和报告；Reasoner 用于需要深度推理的假设生成和因果分析 |
| 数据处理 | pandas (DataFrame) | 纯内存操作，无需数据库，用户上传即用 |
| 前端 | Streamlit | 快速原型，聊天式交互 |
| 可观测性 | 自建 TraceLog | 记录每个节点的输入/输出/耗时/错误，支持 JSON 导出 |

### LLM 双模型策略

系统使用两种 DeepSeek 模型，按任务特点分配：

| 模型 | 用于 | 调用频率 | 理由 |
|------|------|---------|------|
| deepseek-chat | 代码生成、精准查询、报告 | 高（循环中调用） | 速度快、成本低 |
| deepseek-reasoner | 假设生成、因果推理 | 低（每次分析最多 1-2 次） | 推理能力强，但速度慢、成本高 |

**设计考量**：v5.7 之前假设生成和精准查询都用 reasoner，导致循环中频繁调用高成本模型。拆分后，reasoner 仅在需要"想新角度"时调用 1 次，循环内的查询生成全部用 chat。

## 版本演进

| 版本 | 核心变化 | 解决的问题 |
|------|---------|-----------|
| v5 | PythonAgent + Skill 系统 | 代码生成准确率提升 |
| v7 | 扫描 + 检测 + 因果推理三步架构 | 从逐假设验证改为全维度扫描 |
| v8 | Commander + Quick Scan + Deep RCA | 统一扫描入口，引入两轮交互 |
| v9 | 统一管线 + 自适应深度 | 消除 simple/complex 路由错误 |
| v9.9 | 死代码清理（-1055 行） | 移除 v5 遗留的废弃路径 |

## 评测概览

12 个评测 Case 覆盖三组能力维度：

| 分析深度 | Case 数 | 测试能力 |
|---------|--------|---------|
| descriptive | 3 | 误报抑制、阈值判断、时间窗口灵活性 |
| diagnostic / causal | 4 | 异常检测、跨维度下钻、多因素归因、因果链推理 |
| causal + Deep RCA | 5 | 信心判断、补充数据建议、深度根因定位 |
