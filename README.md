# Insight Agent — 电商数据异常检测与根因分析

> 上传报表 + 自然语言提问 → 全维度异常扫描 → 因果推理 → 可执行的分析报告

用户上传 CSV/Excel 电商经营数据，用一句话提问（"为什么 8 月 GMV 下降了？"），Agent 自动完成全维度扫描、因果归因、按需深度分析，输出带行动建议的结构化报告。

---

## 为什么做这个产品

### 用户是谁

中小电商运营者。他们每天盯着后台数据，能直觉感知到"哪里不对"——GMV 掉了、转化率跌了、某个品类突然卖不动了。**他们天然会提出问题，但缺乏系统性定位根因的能力。**

### 痛点在哪

根因分析的门槛不在工具，而在两件事：

1. **提出正确的问题** — 运营者知道"GMV 跌了"，但不知道该拆哪些维度、看哪些指标来定位原因。专业的数据分析师会设计下钻路径（品类→渠道→时段→客群），但中小电商团队往往没有这个角色。
2. **提供恰当的数据** — 即使知道要分析，也不清楚该拉哪些表、补充什么维度的数据才能得到有效结论。

这两个门槛把大量中小运营者挡在了"数据驱动决策"之外。他们手里有数据，但数据的价值没有被释放。

### 商业假设

具备根因分析能力可以直接指导运营动作（调整投放、优化品类结构、定位转化瓶颈），**对收入有直接影响**。对于中小电商而言，这项能力原本需要雇人或外包，成本高、周期长。如果 AI 能把门槛降到"上传一张表 + 问一句话"，用户的付费意愿会更高——因为他们清楚这件事的价值，只是以前做不了。

---

## 产品如何解决这两个门槛

核心思路：**不要求用户具备分析能力，而是把分析能力嵌入流程本身。**

### 三步渐进式架构 — 每一步都在降低门槛

```
Step 1: Quick Scan（单表即可启动）
  用户只需上传一张经营汇总表 + 一句提问
  → Agent 自动完成全维度异常扫描
  → 输出: 哪些指标异常、异常程度、初步归因方向
  → 如果问题简单（"有没有异常？"），到这里就结束

     ↓ 问题需要更深分析时，自动进入

Step 2: 因果推理
  → 基于扫描结果构建因果链（转化率↓ → 买家数↓ → GMV↓）
  → 输出: 根因定位 + 置信度判断
  → 如果数据充分，到这里就结束

     ↓ 置信度不足时

Step 3: Deep RCA（引导式补充数据）
  → Agent 明确告知用户: "需要什么数据、什么格式、为什么需要"
  → 用户上传补充数据后，自动合并扫描 + 假设验证
  → 输出: 综合分析报告 + 可执行的行动建议
```

**关键设计决策：**

- **从单表出发，而非要求用户一次性准备完整数据** — 降低启动门槛，让用户先看到价值再投入更多。
- **每一步都可以独立结束** — 简单问题不过度分析，用户不会被迫走完全流程。
- **Step 1 解决"提出正确的问题"** — 用户不需要知道该拆哪些维度，Agent 自动做全维度扫描，替用户发现问题。
- **Step 3 解决"提供恰当的数据"** — 不说"请补充相关数据"，而是具体指出需要什么表、包含哪些字段，把模糊需求变成可执行的动作。

### 未来延伸：Step 1 的自动化

Step 1（Quick Scan）本身是一个轻量、确定性高的能力模块。后续计划将其独立为**定时自动执行的监控 Agent**：每天/每周自动扫描最新数据，检测异常，只在发现值得关注的变化时通知用户并询问是否进入深度分析。这样用户连"提问"这一步都不需要了——Agent 主动发现问题。

## Architecture

```
用户提问 + 上传数据
       ↓
  Gateway（合法性校验）
       ↓
  Step 1: Quick Scan
  Commander(LLM) → 全维度扫描(L1→L2) → 异常检测(纯规则) → Arbiter(仅不确定时)
       ↓
  按分析深度自动分流 ← Commander 从问题语义判断，非关键词匹配
       │
       ├─ descriptive "有没有异常？"  → 扫描结果报告 → END        (2-3 次 LLM, ~120s)
       ├─ diagnostic  "哪里出了问题？" → + 因果推理 → 报告 → END   (3-5 次 LLM, ~120-180s)
       └─ causal      "为什么下降？"   → + 因果推理 → 可选 Deep RCA (3-12 次 LLM, ~180-360s)
                                              ↓ 信心不足时
                                         展示初步结论 → 用户上传补充数据（可选）
                                              ↓
                                         Deep RCA: 全量扫描 + 假设验证 → 综合报告
```

> 详细架构设计见 [architecture/system_overview.md](architecture/system_overview.md)

## Demo

### 归因分析（causal 路径）

```
提问: "为什么8月美妆付费推广转化率下降了？"

→ Commander: causal 深度, 目标指标=支付转化率
→ Quick Scan L1: 14个指标环比扫描, 发现3个异常
→ Quick Scan L2: 按品类×流量来源下钻
→ Reasoner: 美妆-付费推广转化率 -55%, 因果链: 转化率↓→支付买家数↓→GMV↓
→ Reporter: 报告 + P0行动建议

LLM调用: 4次
```

### 零异常智能响应

```
提问: "淘宝投放效率为什么变差了？"

→ Quick Scan: 0个异常
→ Reasoner: 检测到 question_category=roi_ad, 当前数据无投放明细列
→ 输出: "现有汇总数据各指标正常, 建议补充投放渠道明细数据做深度分析"
   (而非简单说"一切正常")
```

> 这个 case 体现了产品设计原则：即使数据不足以得出结论，也要告诉用户**缺什么、怎么补**，而不是给一个无用的"一切正常"。

## Design Principles

- **Schema 零硬编码** — 指标、维度、聚合方式从数据动态推断，换数据无需改代码
- **LLM 调用最小化** — 异常检测纯规则；补充数据扫描纯 pandas；descriptive 全流程仅 2-3 次调用
- **渐进式深度** — 每一步只在必要时进入下一层，避免简单问题过度分析
- **Fallback 安全网** — Commander 失败 → 规则兜底；Quick Scan 失败 → ReAct 降级
- **Human-in-the-loop** — Deep RCA 需用户确认，AI 不在数据不足时编造根因
